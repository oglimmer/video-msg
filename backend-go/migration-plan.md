# Migration Plan: vmsg Spring Boot → Go

## 1. Application Overview

A video message recording tool backend that accepts WebM video uploads, re-encodes them asynchronously via FFmpeg (VP9 + Opus), stores them on disk with date-based directory structure, and streams them back. Uses MariaDB for metadata storage.

**Current stack:** Spring Boot 4.0.3, Java 21, JPA/Hibernate, MapStruct, Lombok
**Target stack:** Go with chi router, database/sql, go-sql-driver/mysql

## 4. Business Logic Components

| Spring Service | Go Equivalent | Responsibility |
|---|---|---|
| `RecordingService` | `internal/service/recording.go` | Upload orchestration, fetch metadata, get stream resource |
| `FileStorageService` | `internal/service/filestorage.go` | Save/retrieve files on disk, date-based directory structure (`YYYY/MM/DD/{uuid}.ext`) |
| `VideoProcessingService` | `internal/service/videoprocessing.go` | Async goroutine: trigger re-encoding, update DB status |
| `VideoReencodingService` | `internal/service/reencoding.go` | FFmpeg exec: re-encode to VP9+Opus WebM |

### FFmpeg command (must match exactly):
```
ffmpeg -i input -c:v libvpx-vp9 -b:v 1M -crf 31 -maxrate 1.5M -bufsize 2M
       -c:a libopus -b:a 128k -vbr on -f webm -avoid_negative_ts make_zero output
```

**Schema management:** Hibernate `ddl-auto: update` (no Flyway migrations)

| Column | Type | Constraints |
|--------|------|-------------|
| id | BIGINT | PK, AUTO_INCREMENT |
| uuid | VARCHAR(36) | UNIQUE, NOT NULL |
| filename | VARCHAR(255) | |

| file_size | BIGINT | |
| content_type | VARCHAR(100) | |
| duration | BIGINT | NULLABLE |
| processing_status | VARCHAR(255) | ENUM string |
| processing_error | VARCHAR(500) | NULLABLE |
| created_at | DATETIME(6) | |

**Go approach:** Use `database/sql` with raw SQL. Create the table via a migration SQL file run at startup (replacing Hibernate ddl-auto). Use `golang-migrate/migrate` for schema management.

### Queries needed:
- `INSERT INTO recording (...) VALUES (...)` — after upload
- `SELECT * FROM recording WHERE uuid = ?` — fetch by UUID
- `UPDATE recording SET ... WHERE id = ?` — update after re-encoding

## 11. Dockerfile Plan

```dockerfile
# Build stage
FROM golang:1.23-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd/server

# Runtime stage
FROM alpine:3.20
RUN apk --no-cache add ca-certificates tzdata ffmpeg
RUN addgroup -g 10001 appgroup && adduser -u 10001 -G appgroup -D appuser
WORKDIR /app
COPY --from=builder /app/server .
COPY --from=builder /app/migrations ./migrations
RUN mkdir -p /app/video-storage && chown appuser:appgroup /app/video-storage
USER appuser
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=10s --retries=3 CMD wget -qO- http://localhost:8080/api/actuator/health || exit 1
CMD ["./server"]
```

Key differences from Spring version:
- No JVM = much smaller image, faster startup
- FFmpeg still required (apk add ffmpeg)
- Same non-root user pattern (10001)
- Same health check endpoint

## 12. Known Risks & Edge Cases

1. **Async processing:** Spring `@Async` → Go goroutine. Must ensure goroutine doesn't leak on shutdown (use context cancellation).
2. **Multipart parsing:** Go's `multipart.Reader` handles large files differently than Spring. Need to set `http.MaxBytesReader` or not limit (Spring config is `-1` = unlimited).
3. **Date/time formatting:** Spring's `LocalDateTime` serializes as `"2025-01-01T00:00:00"` (no timezone). Go must use `time.Format("2006-01-02T15:04:05")` to match.
4. **Content-Type parsing:** The Spring controller strips codec parameters if `MediaType.parseMediaType` fails. Replicate this fallback.
5. **Graceful shutdown:** Must wait for in-flight re-encoding goroutines before exiting.
6. **File size limits:** Spring sets `max-file-size: -1` (unlimited). Go handler should not impose a limit.
7. **Health endpoint:** Must serve `/api/actuator/health` returning `{"status":"UP"}` for Docker/K8s compatibility.
